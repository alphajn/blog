开始 #

为什么要使用这种方式 #

  1. 解决包相互引用一包升级其他依赖也要升级
  2. 配置重复性问题（编译器、代码格式化、配置文件、测试集成）
  3. 格式不统一，没有统一的文档可供查看

命令行 #

  • pnpm i 根目录下更新所有的依赖
  • pnpm dev 开发运行时
  • pnpm test 运行单元测试
  • pnpm test:ui 运行可视化单元测试
  • pnpm coverage 测试覆盖率
  • pnpm release 发布npm包
  • pnpm lint 代码检查

安装依赖命令 #

  • pnpm add -Dw [test] 全局安装开发依赖 -D === devDependencies
  • pnpm add [bignumber.js] --save-peer --filter @fe/number 安装外部依赖到指定包添加为 平行/开发 依赖
  • pnpm add [@fe/number] --filter @fe/test 安装内部依赖到指定包

项目相关依赖 #

  • pnpm 包管理工具
  • vitest 测试工具
  • vitepress 文档管理
  • bumpp 包发布管理
  • rollup 模块打包器

开发指南 #

命令 #

  1. pnpm i 更新依赖
  2. pnpm dev 开发模式

新增包(分类包或单独包) #

  • 分类包： 指一个包下有n多个模块，所有模块会有统一出口。例如：@fe/utils 里有 foo 和 bar ... 模块
  • 单独包： 指单独一个系列模块，单独打一个包。例如：@fe/number 里有 lt、gt、times 等方法

模块包含文件 #

├─ foo  模块名称
│  ├─ index.ts  入口文件
│  ├─ index.md  模块使用文档
│  ├─ index.test.ts 单元测试
│  └─ demo.vue  使用demo(非必需)

分类包, 例如：@fe/utils #

  1. packages/utils 创建文件夹

  2. packages/utils/foo 添加模块

  3. 模块下添加模块所包含的文件

  4. packages/utils/README.md npm展示文档

  5. packages/utils/index.ts 统一入口文件（非必需）,包含所有的模块. 例如：

    export * from './foo';
export * from './bar';

  6. packages/utils/package.json 配置文件

npm包管理 #

  • pnpm adduser --registry http://localhost:4873/ 注册
  • npm login 登录
  • pnpm release 发布包
  • pnpm unpublish [@fe/number] --force 撤销发布的包
  • git tag -d $(git tag -l) 批量删除本地tag
  • git push origin --delete $(git tag -l) 批量删除远程tag
